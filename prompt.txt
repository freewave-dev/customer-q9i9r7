FREEWAVE.ONLINE — SPEED OPTIMIZATION PROMPT
=============================================

REVIEWER'S HONEST ASSESSMENT OF THE SPEED REPORT
------------------------------------------------
Some of the report is useful. Some of it is boilerplate that does not apply
to this site. Here is the plain-English breakdown before the actionable prompt:

What does NOT apply to freewave.online:
  - "Third-party scripts blocking hydration" — our site is static HTML/CSS,
    there is no framework to hydrate. No React, Vue, etc.
  - "Code-splitting strategy" — we have no JS bundle to split. Our only JS
    is a tiny nav.js (~60 lines) and the Cloudflare Turnstile widget on form
    pages (already loaded with async/defer).
  - "INP 280ms / 320ms" — INP measures input-to-next-paint responsiveness,
    which is tied to JavaScript execution. On a near-JS-free site like ours,
    INP numbers that high are suspicious. It's unlikely the real world
    experience is that sluggish on click/tap.
  - "WebP/AVIF conversion" / "compressing assets" — we don't control the
    asset pipeline from this repo. Images live at media.freewave.dev (R2).
    Converting to WebP/AVIF would need to happen at the bucket level, not
    here. We CAN, however, reference a .webp URL if one exists.

What DOES apply and is worth doing:
  1. LCP image priority. On pages that have a hero/large image near the top
     (e.g. backup.html, waitlist-thank-you.html, support-plan.html,
     demo.html), add fetchpriority="high" to the first above-the-fold <img>.
  2. Lazy-load below-the-fold images. Add loading="lazy" + decoding="async"
     to any <img> that is not visible on initial load.
  3. Explicit width/height on every <img>. Most already have this, but a
     few use inline style="max-width:..." without width/height attributes.
     Explicit dimensions prevent layout shift (helps CLS, which is already
     fine at 0.08 but worth keeping).
  4. Preload the LCP image in <head> on pages where it's a known image —
     e.g. <link rel="preload" as="image"
     href="https://media.freewave.dev/q9i9r7/images/Peplinkmultiwan.jpeg"
     fetchpriority="high"> in backup.html.
  5. Preconnect to media.freewave.dev early. Add
     <link rel="preconnect" href="https://media.freewave.dev" crossorigin>
     in the <head> of pages that load images from there. Saves DNS/TLS time.
  6. Home page LCP is the hero h1 text (hero uses a CSS gradient, no image).
     Text LCP at 3.8s on mobile points at slow TTFB or CSS blocking, not
     images. Cloudflare Pages is usually fast, so this may be a measurement
     artifact. Real-user monitoring (Cloudflare Web Analytics) would confirm.

ACTIONABLE PROMPT (copy/paste this for your next change request)
----------------------------------------------------------------
Do a lightweight, no-JavaScript performance pass on the site. Make only
these changes:

1. On every page that has an <img>, audit each image tag:
   - If it's above-the-fold (hero, first screen): add fetchpriority="high"
     to the single most prominent image only.
   - If it's below-the-fold: add loading="lazy" and decoding="async".
   - If it's missing width and height attributes, add them (measure from
     the image's natural dimensions).

2. On pages that load images from media.freewave.dev, add the following
   to the <head> (just after the <meta viewport> line):
     <link rel="preconnect" href="https://media.freewave.dev" crossorigin>
     <link rel="dns-prefetch" href="https://media.freewave.dev">

3. On pages with a clear LCP image (backup.html, waitlist-thank-you.html,
   support-plan.html, demo.html, calyx-internet.html), also add:
     <link rel="preload" as="image" href="<same-url-as-the-LCP-img>"
      fetchpriority="high">

4. Do NOT add any <script> tags, inline JS, or change nav.js.
5. Do NOT add external CDN resources beyond the media.freewave.dev,
   unsplash, and Google Fonts domains already allowed in CLAUDE.md.
6. Do NOT try to convert images to WebP/AVIF — that's a bucket-level task.
7. After changes, commit and push. Keep the diff focused.
